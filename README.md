# 💰 Учёт финансов Школы (ИШИТР)

Веб-приложение для учёта финансово-хозяйственной деятельности ИШИТР, заменяющее сложные электронные таблицы. Система обеспечивает полуавтоматическую категоризацию затрат по видам деятельности и подразделениям, расчёт весов распределения и формирование аналитических отчётов. Это не бухгалтерская система, а аналитический инструмент поверх фактических данных о тратах.

## Предпосылки

Финансово-хозяйственный учёт ИШИТР ведётся в сложных электронных таблицах, что приводит к ошибкам, трудоёмкости обновления и затруднённому анализу. Ручная категоризация затрат по подразделениям и видам деятельности отнимает значительное время. Отсутствует инструмент для автоматизированного расчёта весов распределения и формирования наглядных аналитических отчётов.

### Как работает текущий процесс

```
Полный перечень трат (суммы, даты, источники)
    ↓
Уполномоченный расставляет теги (отделение, вид деятельности)
    ↓
Расчёт весов — пропорциональное распределение по тегам
    ↓
Итоговый свод по видам деятельности и подразделениям
```

Задача проекта — автоматизировать этот процесс в удобном веб-интерфейсе.

## Цели проекта

1. Реализовать импорт данных о затратах из Excel-файлов
2. Разработать интерфейс тегирования затрат по отделениям и видам деятельности
3. Реализовать алгоритм автоматического расчёта весов и распределения затрат
4. Создать аналитические дашборды с визуализацией данных по различным срезам
5. Обеспечить экспорт итоговых сводных отчётов

## Как мы работаем с Git

### Основные правила

Мы используем **Git Flow** — модель ветвления, которая помогает организовать совместную разработку и не ломать рабочий код.

### Ветки

| Ветка | Назначение |
|-------|-----------|
| `main` | Стабильная версия. Сюда попадает только проверенный и протестированный код. Напрямую в `main` **никто не коммитит** |
| `develop` | Основная ветка разработки. Сюда вливаются все завершённые задачи. Из неё собираются релизы в `main` |
| `feature/*` | Ветки под конкретные задачи. Создаются от `develop`, вливаются обратно в `develop` через Pull Request |
| `fix/*` | Ветки для исправления багов. Аналогично `feature/*` |

### Как работать с задачей

```
1. Берёшь задачу на доске YouGile → переводишь в «В работе»

2. Создаёшь ветку от develop:
   git checkout develop
   git pull origin develop
   git checkout -b feature/название-задачи

3. Работаешь в своей ветке, делаешь коммиты:
   git add .
   git commit -m "feat: добавлен импорт xlsx через SheetJS"

4. Когда задача готова — пушишь ветку и создаёшь Pull Request:
   git push origin feature/название-задачи
   → Создаёшь PR из feature/название-задачи в develop

5. PR проходит код-ревью от лида (куратора) → после одобрения вливается в develop

6. Переводишь задачу на доске в «Готово»
```

### Правила именования веток

```
feature/excel-import        — новая функциональность
feature/weight-algorithm    — новая функциональность
feature/dashboard-charts    — новая функциональность
fix/weight-calculation-bug  — исправление бага
refactor/expense-model      — рефакторинг без изменения поведения
docs/algorithm-description  — документация
```

### Правила коммитов

Используем [Conventional Commits](https://www.conventionalcommits.org/ru/) — это упрощает чтение истории и автоматизацию:

```
feat: реализован парсинг xlsx через SheetJS
feat: добавлен алгоритм расчёта весов по тегам
fix: исправлен расчёт пропорций при множественных тегах
docs: описан алгоритм развесовки в docs/algorithm.md
style: форматирование компонента DashboardChart
refactor: вынесена логика расчёта весов в отдельный сервис
test: добавлены тесты для алгоритма развесовки
chore: обновлены зависимости Recharts
```

**Формат:** `тип: краткое описание на русском`

### Правила Pull Request

1. **Название PR** — кратко описывает что сделано (например: `feat: импорт затрат из Excel`)
2. **Описание** — что сделано, как проверить, скриншоты (если есть UI)
3. **Один PR = одна задача** — не смешивайте несколько задач в одном PR
4. **PR ревьюит лид (куратор)** — без одобрения не мёржим
5. **Конфликты** — автор PR разрешает конфликты самостоятельно, при необходимости обращаясь за помощью

### Чего делать нельзя

- ❌ Коммитить напрямую в `main` или `develop`
- ❌ Делать `force push` в общие ветки
- ❌ Мёржить свой PR без ревью
- ❌ Оставлять ветку без PR дольше 3 дней
- ❌ Коммитить сгенерированные файлы (`node_modules/`, `dist/`, `.env`)
